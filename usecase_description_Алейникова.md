# Диаграмма прецедентов: Система ветеринарной клиники

## Описание предметной области
Система ветеринарной клиники — это программное обеспечение для автоматизации работы ветеринарного учреждения. Она позволяет записывать пациентов на приём, вести медицинские карты животных, назначать лечение, взаимодействовать с лабораторией и формировать отчётность.

## Акторы
- **Владелец животного** — клиент клиники, записывает питомца на приём и получает результаты.
- **Ветеринар** — врач, проводящий осмотр, назначающий лечение и оформляющий рецепты.
- **Администратор** — сотрудник, управляющий расписанием, отчётами и оплатой.
- **Лаборатория** — внешняя система, принимающая анализы и предоставляющая результаты.

## Прецеденты
- **Запись на приём** — выбор времени и врача для визита.
- **Проведение осмотра** — фиксация состояния животного.
- **Назначение лечения** — указание препаратов и процедур.
- **Оформление рецепта** — создание рецепта на лекарства.
- **Сдача анализов** — направление биоматериала в лабораторию.
- **Получение результатов** — просмотр результатов анализов.
- **Управление расписанием** — редактирование графика работы врачей.
- **Ведение карты пациента** — хранение истории болезни животного.
- **Формирование отчёта** — создание отчётов по посещениям и доходам.
- **Оплата услуг** — проведение оплаты за приём и процедуры.

## Отношения
- **include** — проведение осмотра, назначение лечения и сдача анализов требуют доступа к карте пациента.
- **include** — запись на приём невозможна без проверки расписания.
- **extend** — рецепт может быть оформлен только после назначения лечения.
- **extend** — получение результатов возможно только после сдачи анализов.

## Контрольные вопросы

---

### 1. Что такое диаграмма прецедентов и для чего она используется?

**Диаграмма прецедентов (Use Case Diagram)** — это тип диаграмм UML, который описывает функциональные требования к системе с точки зрения пользователя.

**Используется для:**
- Определения границ системы и её взаимодействия с внешним миром
- Выявления основных функций (прецедентов), которые система предоставляет пользователям
- Демонстрации ролей (акторов) и их доступа к функциям
- Согласования требований между заказчиком и разработчиками

---

### 2. Чем актор отличается от прецедента?

| Характеристика | Актор | Прецедент |
|----------------|-------|-----------|
| **Сущность** | Внешняя по отношению к системе | Внутренняя функция системы |
| **Роль** | Взаимодействует с системой | Выполняется системой |
| **Обозначение** | Человечек (стикман) | Эллипс |
| **Пример** | Студент, Преподаватель | «Запись на курс», «Просмотр оценок» |
- **Актор** инициирует действие
- **Прецедент** — это само действие, которое система выполняет в ответ

---

### 3. Что обозначает системная граница?

**Системная граница** — это прямоугольник, который отделяет систему от внешнего мира.

- **Внутри границы** — прецеденты (то, что система делает сама)
- **Снаружи границы** — акторы (пользователи и внешние системы)

**Значение:** Показывает, где заканчивается ответственность системы и начинаются внешние сущности.

---

### 4. В чём разница между отношениями include и extend?

| Характеристика | `<<include>>` | `<<extend>>` |
|----------------|---------------|---------------|
| **Обязательность** | Обязательное выполнение | Выполняется только при условии |
| **Зависимость** | Базовый прецедент **всегда вызывает** включаемый | Расширяющий прецедент **может быть вызван** при условии |
| **Направление стрелки** | `A ..> B` (A включает B) | `A <.. B` (B расширяет A) |
| **Смысл** | «Для выполнения A нужно B» | «Иногда при выполнении A может выполняться B» |
| **Пример** | Запись на курс `<<include>>` Авторизация | Запись на курс `<..` Прохождение тестов |

### 5. Почему Mermaid не подходит для построения полноценных use case диаграмм?

**Основные недостатки Mermaid:**

| Проблема | Пояснение |
|----------|-----------|
| **Нет поддержки `<<include>>` и `<<extend>>`** | Нельзя отобразить эти важные отношения между прецедентами |
| **Нет системной границы** | Нельзя визуально отделить систему от внешнего мира |
| **Ограниченное позиционирование** | Сложно управлять расположением элементов |
| **Стандарт UML** | Mermaid не полностью соответствует нотации UML для use case |

**Вывод:** Mermaid подходит для простых блок-схем, но для профессиональных use case диаграмм лучше использовать PlantUML.

### 6. Какой синтаксис PlantUML используется для определения актора и прецедента?

**Актор:**
```plantuml
actor "Имя актора" as псевдоним
actor Студент as student
```

**Прецедент:**
```plantuml
usecase "Название прецедента" as псевдоним
usecase "Просмотр курсов" as UC1
```

**Связь актора с прецедентом:**
```plantuml
student --> UC1
```

---

### 7. Как в PlantUML задать отношение include?

**Синтаксис:**
A ..> B : <<include>>

**Пример:**
```plantuml
UC2 ..> UC9 : <<include>>
```

**Что означает:** Прецедент A обязательно включает прецедент B. Без B выполнение A невозможно.

**Направление стрелки:** От включающего к включаемому.

---

### 8. Какие правила именования акторов и прецедентов вы знаете?

| Элемент | Правила | Пример |
|---------|---------|--------|
| **Актор** | Имя существительное, отражающее роль; с заглавной буквы; в кавычках | `"Студент"`, `"Преподаватель"`, `"Администратор"` |
| **Прецедент** | Глагольное словосочетание (действие); в кавычках | `"Просмотр курсов"`, `"Запись на курс"` |
| **Псевдонимы** | Латиница, краткие, осмысленные | `as student`, `as UC1` |
| **Системная граница** | Имя системы — существительное в кавычках | `rectangle "Система онлайн-обучения"` |

**Общие правила:**
- Имена должны быть понятны заказчику (не технический жаргон)
- Прецедент начинается с глагола (Просмотр, Запись, Создание)
- Актор — кто выполняет действие
- Для кириллицы обязательны кавычки: `"Текст"`
